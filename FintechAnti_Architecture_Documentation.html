1. Introduction

Financial transaction processing is fundamentally challenged by the massive volume of unstandardized data, the complexity of dynamic taxation slabs (GST at 0%, 5%, 18%, 40%), and an inherently high rate of human error. Traditional compliance workflows rely on post-facto manual effort, creating significant bottlenecks and exposing organizations to unmitigated anomaly risks.

Auditron is an AI-driven intelligent system engineered to automate and secure these financial workflows. The platform operates as a decision intelligence platform (not an accounting system), designed to augment existing data pipelines with machine learning inference.

Key capabilities include:

  • Automated GST Classification: Dynamic, schema-aware mapping of disparate transaction records into defined GST tax slabs.
  • Multivariate Anomaly Detection: Ensemble-based detection of both structural outliers and semantic anomalies.
  • Time-Series Forecasting: Predictive modeling of transaction volumes and category distributions.
  • Human-in-the-Loop Review: Native feedback mechanisms that capture human corrections for continuous model improvement.

4. Detailed Module Architecture

Module 1 – Orchestrator / System Flow

The system orchestrator functions as the primary nervous system for the platform, ensuring correct serialization of tasks across distributed components without maintaining heavy state overhead.

  • Route API calls: Direct incoming web traffic from the frontend to the appropriate microservice endpoints.
  • Manage pipeline flow: Sequence asynchronous operations (e.g., waiting for classification before anomaly detection).
  • Maintain state: Track temporary file paths, job IDs, and system metadata during batch execution.

Module 2 – Classification Architecture

This module forms the primary inference engine for GST categorization, designed to handle highly varied inbound CSV structures via schema hashing.

flowchart LR ING["Data Ingestion"] --> SCH["Schema Detection"] SCH --> FE["Feature Engineering"] FE --> INF["ML Classification"] INF --> OUT["Output Generation"]
  • Detect schema (A/B/C/D/H): Route data to the proper preprocessing pipeline based on column signatures.
  • TF-IDF + Random Forest: Extract textual features and predict GST slab (0%, 5%, 18%, 40%).
  • Confidence scoring: Output probability metrics to trigger manual review if below threshold.

Module 3 – Anomaly Detection Architecture

The anomaly system uses an ensemble approach to cover both numerical deviations and semantic textual mismatches.

flowchart TD DATA["Batch Data"] --> ND["Numeric Detector
(Isolation Forest)"] DATA --> TD["Text Detector
(Embeddings)"] DATA --> AE["Confidence Analyzer
(Autoencoder)"] ND & TD & AE --> ENS["Ensemble Scoring Agent"] ENS --> FLAG["Anomaly Flagging"]
  • Isolation Forest: Identify statistical outliers in multi-dimensional numeric space.
  • Autoencoder: Detect complex reconstruction errors indicating structural deviations.
  • Embedding similarity: Flag semantic anomalies in unstructured text using Sentence Transformers.
  • Adaptive thresholding: Dynamically adjust sensitivity based on historical batch distributions.

Module 4 – Analytics & Decision Layer

This layer converts raw prediction metadata into actionable business intelligence, interfacing directly with the user.

  • KPI visualization: Render metrics for processed entries, anomaly rates, and latency.
  • Human validation: Accept human overrides and record deltas for ground-truth updates.
  • Time-series forecasting: Project future volume and anomaly load using Prophet/ARIMA.
  • Insight generation: Provide explainable contexts for flagged transactions.

Module 5 – Retraining & MLOps Architecture

The MLOps module ensures models adapt to shifting transaction patterns captured via human feedback.

  • Dataset aggregation: Compile raw data and human corrections into retraining corpora.
  • Model training: Execute scheduled refitting of Random Forest models.
  • Evaluation (Macro-F1): Score candidate models against standard validation matrices.
  • Promotion logic: Conditionally advance models to 'Production' in MLflow if metrics exceed baseline.

5. Data Flow / Pipeline

The end-to-end lifecycle of a data batch follows a strict deterministic sequence:

6. Key Design Principles

  • Microservices separation: Hard boundaries between logic domains to decouple scaling constraints.
  • Stateless services: API servers do not maintain local memory state between requests.
  • File-based data exchange: Heavy data matrices exchanged via volume mounts, avoiding memory saturation on transport layers.
  • Human-in-the-loop: Architecture treats human override as a first-class citizen for mitigating model drift.
  • Model lifecycle management: Strict promotion gates to prevent overfit model graduation.

7. Conclusion

Auditron achieves a scalable fintech AI platform via the modular composition of modern distributed systems and robust machine learning pipelines. By operating independently of core accounting structures, it provides an auxiliary decision intelligence layer tailored for rigid real-world compliance workflows.
